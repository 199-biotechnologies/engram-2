## Key Decisions Made

### Architectural choices
- **MAB `haystack_sessions` as opaque `serde_json::Value`** — the field varies wildly by split (flat turns vs triple-nested sessions) and we don't use it downstream. Strict parsing was more trouble than it was worth.
- **LoCoMo-Plus cue as pseudo-session** — we index the cue dialogue as a new synthetic session with id `cue_for_<fnv1a_hash>`, inserted at `query_time - time_gap_days` on the timeline. Trigger is NOT in the haystack (it's the query). The reranker's job is to surface the cue when the trigger is queried. This diverges from the Python `build_context` which stitches BOTH cue and trigger into the dialogue, but makes sense for RAG where trigger is the question.
- **Cognitive judge is binary** — no partial credit. Matches the paper exactly.
- **MAB question_types defaulting to STANDARD** — the real JSONL has `question_types: None` in most rows. We default to STANDARD judge variant when missing.

### Why delegate to Codex instead of implementing directly
- Boris explicitly asked: "get codex to review and implement it for us, and we triple check"
- Codex wrote the lion's share (~2500 lines) but all 4 implementation commits went through human review and test verification at each step

### Why revert the rustfmt drive-by
- Boris's global rule: "Don't add features, refactor, or introduce abstractions beyond what the task requires"
- The fmt changes were pure whitespace, reversible, and didn't affect semantics — but reformatting 35 files that weren't in scope violated the principle
- Kept scoped files' fmt changes since they had real content mixed in

### Why smoke tests used the first 30 of each dataset
- LoCoMo-Plus: first 101 entries are ALL causal. Diverse sampling would require running 201+ questions (expensive). Deferred.
- MAB AR: first row alone has 100 questions (`ruler_qa1_197K`). Running 30 questions stays within one row. Other rows would need new runs.
- Boris acknowledged this is not representative — GPT Pro review explicitly asks about the bias.

## Gotchas & Warnings

### Build and tooling
- **`GEMINI_API_KEY` env var is expired.** Always prefix bench commands with `unset GEMINI_API_KEY` to fall back to the config-file key (ending `lZ1l9M`). The previous handoff warned about this too — it's still true.
- **Never use Python for benchmark execution.** The `scripts/download_memoryagentbench.py` is a one-time dataset converter. The actual bench loop MUST stay in Rust.
- **The zsh wrapper may silently drop large Codex prompts.** If you delegate to Codex with `"$(cat prompt.md)"` argv expansion, the process will hang. Use `codex exec ... - < prompt.md` stdin redirection for any prompt >5KB.

### Benchmark specifics
- **MAB `haystack_sessions` shape varies by split.** Accurate_Retrieval rows from LongMemEval source have `Vec<Vec<Vec<Turn>>>`. Other splits have `Vec<Turn>` or null. The opaque `serde_json::Value` in our loader handles all of them. **Do NOT "fix" this to a strict type** — you will hit row 18 of AR and crash.
- **MAB `question_types` is `None` in most real rows.** The Python reference uses the `source` field to infer type, but our Rust impl just defaults to `STANDARD` when missing. That's fine for now but may be the cause of some mis-judged answers.
- **LoCoMo-Plus 30q is all causal.** The first 101 entries of `locomo_plus.json` are causal, then 100 state, 100 goal, 100 value. Any run with `--limit < 101` will be single-relation-type. Use `--limit 201` to get causal + state + a bit of goal, or `--limit 401` for all four.
- **MAB AR 30q is all RULER QA1.** The first 100 questions of AR come from the `ruler_qa1_197K` row (a synthetic needle-in-haystack). Use `--limit 500+` to escape into LongMemEval and EventQA rows.
- **Our 43.3% and 96.7% numbers are almost certainly inflated by these row biases.** Do NOT report them as "engram beats SOTA" until diverse-row runs complete.

### Implementation details from Codex's work
- **Cognitive judge prompt has `{evidence}` and `{pred}` literals** that get replaced at runtime via `.replace()`. Be careful not to include those substrings in actual content.
- **`build_stitched_dialogue` uses a non-cryptographic FNV-1a hash** for the cue pseudo-session id (`cue_for_<hex>`). This is stable across runs but not across input changes. Don't expect reproducible IDs if you change the cue text.
- **The cue pseudo-session header format is `[cue_for_<hash> — YYYY-MM-DD HH:MM]`** followed by the dialogue body. The `[session_N — ...]` pattern from `flatten_conversation` is NOT used for cues.
- **`run_locomo_plus_qa` reuses the LoCoMo dataset** — pass both `LocomoPlusDataset` AND `LocomoDataset` to it. Each plus entry stitches into `locomo.samples[i % samples.len()]`.
- **The 370-line `run_locomo_plus_qa` has its own chunking loop** — it does NOT call `run_locomo_qa`. If you change `run_locomo_qa`'s hybrid retrieval code, you probably also need to change the copy in `run_locomo_plus_qa`.

### Previous session gotchas that still apply
- The 50-question slice is NOT random (first 50 questions starting at conv-26). Results don't generalize.
- The lenient judge inflates scores by 18+ points vs strict. NEVER revert to the lenient judge.
- R@5 is currently "fractional" — fraction of gold sessions retrieved. Not "any gold session found." The field is still named `recall_at_5` for backward compat with old checkpoints.
- jina-reranker-v3-mlx exists as a sidecar on port 8766 but is NOT wired into `engram recall` (only bench uses it via `ENGRAM_RERANK_PROVIDER=zerank2` env var).

### Codex-specific gotchas
- **Codex has a strong tendency to run `cargo fmt` across the whole codebase.** We reverted 35 files this session. If you delegate to Codex again, explicitly tell it "Do NOT run cargo fmt or rustfmt. Do NOT touch files outside the listed scope."
- **Codex can hang waiting on API with low CPU usage.** Don't mistake `CPU: 0.0%` for stuck. Check `ps -o etime` — if elapsed is growing but CPU isn't, it's waiting on a long reasoning call.
