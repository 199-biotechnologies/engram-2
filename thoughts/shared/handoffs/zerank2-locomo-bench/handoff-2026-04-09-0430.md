## Optimization sweep results (on M4 Max 64 GB, 20 q × 30 docs = 600 pairs)

| variant | pairs/sec | mean latency | peak RAM | status | why |
|---|---:|---:|---:|:---:|---|
| **v1 bf16 (BEST)** | **28.5** | **1054 ms** | **8.07 GB** | ✅ SHIPPED | yes-token shortcut + `as_linear` |
| v0 baseline | 25.0 | 1200 ms | 8.55 GB | ✅ kept | plain `mlx_lm` forward, full vocab projection |
| v2 mx.compile(shapeless) | 26.7 | 1123 ms | 8.00 GB | ❌ regression | compile overhead > fusion gain; mlx-lm already fuses hot kernels |
| v3 Q8 quantization | 24.3 | 1234 ms | 4.69 GB | ❌ regression | dequant cost > bandwidth savings on 546 GB/s unified memory |
| v2 Q4 quantization | n/a | n/a | 2.26 GB | ❌ **FAILED VALIDATION** | max diff 5.47e-1, top-10 ranks flip; classifier heads amplify quant noise |

**Key lesson:** on M4 Max 64 GB, **quantization is not the lever**. Memory is abundant, bandwidth is already high, and the cross-encoder calibration (`yes_token_logit / 5`) has no averaging safety net to hide quant noise. The real next-step wins are batching-oriented (prefix KV cache reuse, larger batches, async eval pipelining), not bit width reduction.

## Critical engineering decisions made this session

1. **Use `gemini-embedding-2-preview`, not `gemini-embedding-001`.** Boris confirmed and we changed `DEFAULT_MODEL` in `crates/engram-embed/src/gemini.rs` + added `GEMINI_EMBED_MODEL` env var override in `from_env()`. New binary installed at `~/.cargo/bin/engram` **post-commit d04ede0** — the currently-running bench uses it.

2. **Stay at bf16 for the MLX reranker weights.** Do not ship Q4 (failed validation) or Q8 (slower + drift). Any future quant experiment must use a calibration-aware path (GPTQ, AWQ, HQQ+, SqueezeLLM), not post-training affine, AND must be validated via `validate_mlx_reranker.py` before shipping.

3. **Skip `mx.compile` on the outer forward** for this workload. mlx-lm's Qwen3 already uses fused primitives; compile adds overhead without meaningful extra fusion. Gated behind `ENGRAM_ZERANK_MLX_NO_COMPILE=1` for future A/B tests, but OFF by default.

4. **Use `as_linear(hidden_last)` instead of direct row indexing for the yes-token projection.** This keeps the score path identical for bf16 AND quantized variants (QuantizedEmbedding's `.weight` is packed and can't be row-sliced). The speed cost vs direct dot product is < 1 % of total time.

5. **MLX sidecar runs on port 8766, NOT 8765.** The legacy orphan on :8765 is still alive. Using :8766 avoids any interaction.

6. **Test workload in `benchmark_mlx_reranker.py` is 20 queries × 30 docs = 600 pairs.** Deterministic, committed as code, fast enough to iterate on in < 30 seconds per variant.

## What was deferred (the roadmap)

Tracked in the zerank-2-mlx README:

- **v4 query-prefix KV cache reuse** — next big lever, potentially 5-10× on `top_k ≈ 20-50`. Risks: `mlx-lm`'s `make_prompt_cache` may fall back on hybrid-attention models; Qwen3 is pure full-attention so it *should* be safe, needs verification and a cache-copy implementation. This is what the GPT Pro review is supposed to sanity-check before we build it.
- **v5 larger batches** — M4 Max 64 GB can batch 60-240 pairs per forward pass. Simple to measure, might be a big win.
- **v6 `mx.async_eval` pipelining** — overlap prefill and extract across concurrent HTTP requests.
- **v7 mixed-precision quantization** via `mlx_lm.convert --quant-predicate`, only if we ever need the memory for 8 B+ variants.

### 1. Why is the LoCoMo-QA bench running at 0.06 q/sec instead of ~0.37 q/sec like Cohere?

The MLX reranker standalone benchmark hits **28.5 pairs/sec** on a 30-doc batch, so a 5-doc rerank call should take ~175 ms. At 17 s / question, the rerank is definitely NOT the bottleneck.

Candidate suspects, ranked:
- **Gemini embed API rate limiting** — the new key might be on a free-tier quota. The embed call is per-question, so any rate-limit backoff would show up directly.
- **OpenAI rate limiting** on the answerer and judge (both `gpt-5.4-mini`).
- **Synchronous call pattern with no overlap** — engram bench may serialize embed → retrieve → rerank → answer → judge per question.
- **Retry backoff** in the various clients (exponential), eating latency on retries that never log to stdout.

## Gotchas (minimal — see prior handoff for the rest)

- **The `GEMINI_EMBED_MODEL` env var change is in the installed binary but NOT in the git tree.** First thing next session: commit it.
- **The running bench uses the new binary** (post-147c0e9 + post-d04ede0 + post-uncommitted gemini change). Crash would be survivable thanks to the JSONL checkpoint.
- **MLX sidecar is a stateful Python process.** If it dies, the bench dies (the Rust retry logic will give up after 6 retries). You'll have to restart it with:
  ```bash
  uv run --with mlx --with 'mlx-lm>=0.21' --with 'transformers<5.0,>=4.45' \
         --with safetensors --with huggingface_hub --with numpy \
         crates/engram-rerank/python/zerank_server_mlx.py
  ```
  (or equivalently the copy in `~/Projects/zerank-2-mlx/`).
- **`transformers>=5.0` breaks the PyTorch reference** (the `_tied_weights_keys` list handling changed). Pin `<5.0` when running the validator.
- **`sentence_transformers.CrossEncoder.predict()` is incompatible** with `zerank-2`'s `ZeroEntropyTokenizer.__call__` signature in `sentence-transformers >= 3.0`. The upstream Python sidecar uses direct `AutoModelForSequenceClassification` + manual forward. Our MLX port uses `Qwen2TokenizerFast` + inlined chat template.
- **Checkpoint file schema** is one JSON object per line from `run_locomo_qa` in `crates/engram-bench/src/qa.rs`. The `scripts/bench_eta.sh` expects fields `question_type` (or `category`) and `correct`. If jq fails, the schema may have changed — check the first line with `head -1 <file> | jq .` and adjust the script.
