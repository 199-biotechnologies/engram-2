## What's NEW since handoff-1 (a few hours earlier)

### 1. Two more retry fixes shipped
- **`cf64c28`** — openrouter full-lifecycle retry (send + body read + parse + 5xx)
- **`9b08bb0`** — zerank_local full-lifecycle retry (localhost can drop too under contention)
- **`4eccb0d`** — cohere reranker full-lifecycle retry (the fourth retry fix; cohere returned 502 after 36 minutes into the LoCoMo baseline run, killing the bench. This commit adds the same try-loop pattern to `cohere.rs`)

**Every API client engram talks to now has identical retry coverage:** send / 429 / 5xx / body read / JSON parse → retry up to 6 times with exponential backoff. The empirical lesson: at 1500+ sequential requests, EVERY upstream provider hits at least one transient per run.

### 2. zerank-2 sidecar code fixes (commits `edb77b2`, then internal sidecar restart)
- Pin `transformers<5.0,>=4.45` (transformers 5.x broke `_tied_weights_keys` when it's a list)
- Use `AutoModelForSequenceClassification`, NOT `AutoModelForCausalLM` (zerank-2 registers as ForSequenceClassification despite returning CausalLM-style outputs)
- Don't use `sentence_transformers.CrossEncoder.predict()` — the `ZeroEntropyTokenizer.__call__` signature is incompatible. Use direct AutoTokenizer + manual forward + `.squeeze(-1)`.
- The model returns `[batch, 1]` already calibrated (yes-token logit / 5) — don't try to do manual extraction.
- `torch_dtype` is deprecated in transformers 4.46+; use `dtype=` instead.

### 3. Per-question JSONL checkpoint shipped (commit `147c0e9`) ⭐
**Both** `run_locomo_qa` and `run_longmemeval_qa` in `crates/engram-bench/src/qa.rs` now take a `checkpoint_path: Option<PathBuf>` parameter. Each completed question is appended to a `.checkpoint.jsonl` file as a single JSON line and `sync_data()`-ed before the next question runs. Best-effort: a checkpoint write failure is logged but never aborts the bench.

The bench CLI (`run_locomo_qa` and `run_longmemeval_qa` in `bench.rs`) automatically computes `save_path.with_extension("checkpoint.jsonl")` and passes it. Default file naming dropped the question count from the timestamp filename (was `locomo-qa-{ts}-{count}.json`, now `locomo-qa-{ts}.json` — count is no longer known until end of run).

**The currently-running benches do NOT use this code.** They were launched against the pre-`147c0e9` binary already loaded into memory. A new invocation (after `cargo install --path crates/engram-cli --force`) gets the checkpointed behavior.

### 4. The OOM event
At ~01:10 (about 30 min into both benches), the macOS OOM killer fired SIGKILL (exit 137) on the Python sidecar process (PID 63861). Cause: memory pressure from the 4B model held in unified MPS memory + a 5.4GB next-server + many iOS Simulator processes pushed `Pages free` to 4120 (~65 MB). System started compressing 10 GB of memory then SIGKILL'd the largest singletons.

**Recovery sequence:** restarted sidecar via `uv run` immediately → loaded in 12.8s from cache (model files already on disk) → bench's retry logic re-connected within budget → both runs survived.

The model is back in Metal memory now and free RAM is again at 0.1 GB. A second OOM is plausible. The smart move was to kill the next-server (5.4 GB) — Boris was told but I'm not allowed to kill arbitrary user PIDs without explicit consent.

## Decisions made since the last handoff

- **Per-question checkpoint is the right design.** Boris explicitly corrected my earlier interpretation of "don't save on error". His exact words: *"partial benchmark is still useful... that was a fucking stupid decision u did not to save on error"*. I had read the original instruction as "no partial saves at all", but he meant "no corrupt or partial-as-if-complete data". An append-only JSONL is neither: each line is a complete `QaRunResult`, and the file isn't claimed as the final report. This is now committed in `147c0e9`.

- **Don't kill next-server without consent.** Boris said "hey hey dont kill anything" when I suggested he run `kill 85877`. I clarified the suggestion was for him to run, not for me. Still applies for any new session: never run `kill` on arbitrary user PIDs without explicit "do it".

- **Cohere v3.5 actually HURT R@5 by 1pp on engram's LongMemEval data.** This is in our own measurement (HANDOFF.md). zerank-2 is being tested specifically because the published per-domain numbers say it's much better on biomedical/STEM, which is engram's target workload. LoCoMo doesn't have a biomedical category specifically — its 5 categories are single-hop, multi-hop, temporal, open-domain, adversarial. Lift may be smaller in practice than the published deltas.

- **The 28% → 82% number from earlier is on a 50-Q biased sample.** Don't claim engram is at 82% on full LoCoMo until the full bench finishes. The full 1542-question set includes category 4 (open-domain knowledge, 841 questions) which the 50-Q sample completely missed. The full result may be substantially lower.
